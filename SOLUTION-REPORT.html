1.Executive summary

The need. The CI team needs to (a) stay current daily on the software-supply-chain-security industry and (b) see how JFrog compares to its competitors — reliably enough to act on in a deal.

The build. A Retrieval-Augmented Generation (RAG) engine. It collects evidence about 10 vendors from the open web, uses an LLM to compile each source once into structured, cited facts, and stores everything in one Cloud SQL Postgres database that serves as vector store, relationship graph, and freshness ledger. That knowledge is served three ways: a web console with cited chat and on-demand reports, a from-scratch MCP server any AI client can call, and a Telegram assistant.

Why this shape. A plain LLM hallucinates competitor facts and has no concept of freshness. A web-search-at-answer-time tool is slow and un-auditable. This engine separates acquisition (online, earlier) from answering (offline, from the database), so every answer is traceable to a dated source and the database is the single source of truth.

2.The solution & why

The two needs are met by two distinct mechanisms.

Staying current is a discovery + supersession loop, not a re-scan of everything. A source whose text is unchanged is recognized by its SHA-256 hash and skipped. When a newer source covers the same vendor and the same topic, the older one is marked superseded and drops out of retrieval. A gap pass then re-acquires whatever the coverage map shows as missing or stale. (This loop runs on demand today; scheduling it daily is the first roadmap item.)

Comparing JFrog to competitors is a structured lookup, not free-text prompting. Every fact is filed against one of 42 shared dimensions (e.g. container_image_scanning, pricing_packaging) and given a coverage state per vendor. “How does JFrog compare to Snyk on container scanning?” becomes “fetch both vendors’ active evidence on container_image_scanning and report each state” — including the cells where a vendor has no evidence, which are reported as gaps rather than filled in.

Scope: 10 vendors — JFrog (anchor), Sonatype, Snyk, Black Duck, Endor Labs, Checkmarx, Mend, GitLab, GitHub, Aqua Security. Four are deep-mapped now (JFrog, Snyk, Sonatype, GitLab); the rest are seeded and deepen on demand.

3.Design

3.1 Architecture

One rule defines the system: acquisition is the only layer that touches the internet; everything downstream reads only from the database. The architecture diagram above shows the full pipeline; the high-fidelity renderings live in the repo (architecture, infrastructure).

3.2 The database & how data is sorted

What it is. A single Cloud SQL PostgreSQL 16 instance with the pgvector extension. It is three things at once, which is why no second datastore is needed:

GroupTablesHolds
Corpussources, chunksOne row per source; compiled text split into embedded chunks
Graphentities, relationshipsProducts/features/integrations and the typed edges between them
Provenancesource_citationsThe official URLs each source cited
Coveragedimension_coverage_assertions, …_statusPer-source capability claims, and their rollup per vendor×dimension
Auditdimension_coverage_audit, source_healing_auditAppend-only log of every status/label change

How a source is sorted. A web page on its own is just text — to be retrievable for a specific competitive question it has to be filed. “Sorting” here means deciding four things about every source and writing them onto each chunk cut from it, so a question can later ask for exactly the right slice of the corpus.

competitor
Whose evidence is this?
JFrog, Snyk, Sonatype… — set by the lane that searched for it
axis
Product fact or business fact?
technical · business · both — relevance gate, confirmed at synthesis
doc_type
What kind of document?
docs · release_notes · pricing · analyst · blog · news · company_fact
dimension
Which topic does it speak to?
one of 42 canonical dimensions — normalized by ontology.py

So a Snyk container-scanning docs page becomes chunks each tagged competitor=Snyk, axis=technical, doc_type=docs, dimension=container_image_scanning, status=active. doc_type matters for more than filtering: it sets the freshness half-life, so the kind of document determines how long the engine trusts it.

These four keys are copied onto every chunks row rather than kept only on the parent sources row and joined at query time. Joining is tidier in theory, but retrieval is the one path that must be fast, and a vector search over tens of thousands of chunks should not pay for a join on every query. With the keys on the chunk itself, a single composite index narrows the search to “Snyk + technical + container scanning + active” before the expensive vector comparison runs — the database ranks a few hundred candidate chunks instead of the whole table.

Raw-to-processed traceability. Before anything is sorted or embedded, the raw source is captured verbatim — original HTML/markdown snapshotted with a provenance header — and the source row stores a content_hash and a raw_path pointer to it. Every artifact is traceable backwards: a vector belongs to a chunks row → a source_id → the sources row with the original URL, hash, and raw capture. That chain — raw ↔ source ↔ chunk ↔ vector — lets any answer be audited back to the untouched material. (Raw snapshots currently live on the instance filesystem; moving them into managed storage is a roadmap item — §7.)

The status lifecycle

Every source — and every chunk derived from it — carries one of four statuses, and this is the freshness mechanism. Incoming candidates pass a content-hash dedup gate first, so an unchanged page is skipped rather than re-stored:

Three things make this lifecycle trustworthy. First, a source and all of its chunks share one status and move together, so retrieval can never surface an active chunk that belongs to a superseded source. Second, every transition is appended to an audit table (source_healing_audit: old status, new status, reason) and nothing is deleted — healing is non-destructive, so a wrongly-staled source can be restored (the stale → active arrow). Third, the transitions are triggered differently. superseded is automatic on every ingest — a newer Snyk page on container_image_scanning flips the previous one and it instantly drops out of retrieval, which is how “update outdated info with updated info” actually happens. stale is age-driven against a per-doc_type half-life (news 21d, release_notes 60, pricing 90, docs 120, analyst 180, company_fact 365), a source going stale once older than twice its half-life. dead is set when a link stops resolving. Supersession runs today on every ingest; the scheduled age-out and link-check cleaner that drives stale/dead at scale is the freshness job in §7.

The important design point is where freshness is enforced. The pgvector HNSW index is partial — built only over active chunks. Stale, superseded, and dead evidence is not even in the index the search walks, so excluding it costs nothing at query time. In practice the retrieval query is a single SQL statement — WHERE status='active' + metadata filters + ORDER BY embedding <=> :q — in which lifecycle filtering, metadata narrowing, and similarity ranking happen at once, in one system.

3.3 The ontology

What it is. A controlled vocabulary of 42 dimensions — the fixed set of comparable topics the engine tracks — split across two axes: 30 technical (e.g. software_composition_analysis, container_image_scanning, sbom_generation, package_firewall, reachability_analysis, runtime_security, supported_ecosystems) and 12 business (e.g. market_positioning, pricing_packaging, gtm_motion, funding_ownership, customers_case_studies, win_loss_signals). The full list is in Appendix C. It is defined once in config.yaml; nothing in the code hardcodes a dimension.

Why it exists. Comparison only works if both vendors are measured against the same axes. The ontology is that shared yardstick — it turns an open-ended “tell me about Snyk” into a checklist of specific, comparable questions, and it gives the coverage map a fixed set of rows.

How it’s used — three ways, all from the same list:

  1. Research checklist. The deep-map iterates vendor × dimension, acquiring evidence for each cell, so coverage is systematic rather than whatever the web surfaced.
  2. Retrieval filter. dimension is a key on every chunk, so a query can be scoped to exactly the topic asked about.
  3. Coverage-matrix rows. Each vendor × dimension cell holds a coverage state, which is what makes an honest comparison table possible.

Keeping it coherent. Sources describe the same capability in many words. ontology.py normalizes a raw label through four stages — exact match → a ~122-entry alias map (vulnerability_remediation → autofix_remediation) → ~69 context-sensitive patterns (text mentioning “reachab…” → reachability_analysis) → an axis fallback. If none match, the original label is kept rather than forced into a wrong bucket. The system never invents a dimension and never silently mislabels one.

3.4 Acquisition & deep search

This is the only layer that touches the internet, and the quality of everything downstream depends on it. Three complementary lanes feed the pipeline; every candidate they return is normalized to the same shape (title, url, text, vendor, date, source kind) and snapshotted for provenance.

LaneAxisSourceBest for
Web research (deep search)bothClaude + live web searchbroad first-party baseline (cold start)
TavilybusinessTavily search APIpricing, funding, analyst, customers, news
Context7technicalContext7 MCP (indexed docs)current product/SDK documentation

The deep-search lane, in detail. A naïve scraper produces marketing noise and misses the structure a comparison needs. This lane instead treats acquisition as a research task and hands it to a reasoning model. It calls Claude (Sonnet) with the live web-search tool (up to 12 searches) under a skill that imposes one hard constraint: use official, first-party sources only — corporate site, product/developer docs, official blog, press/IR, official GitHub — and explicitly exclude third-party news, analyst sites, forums, review sites, and Wikipedia. The reason is trust: a vendor’s own documentation is the most authoritative statement of what their product does, and keeping third-party interpretation out of the baseline means every fact traces to the vendor itself.

The model returns one long, structured report walking the whole ontology — snapshot, business, market, financials, people, a technical deep-dive (architecture, APIs, SDKs, scalability, differentiation, and explicitly its limitations), product/traction, strategy/roadmap, and a SWOT — with anything undisclosed marked “not officially disclosed” rather than guessed. A cheap splitter (Haiku) then cuts it into ontology-aligned slices, each preserving the official URLs it cited and addressed by a synthetic ci-report://snyk/<hash>#sbom_generation-style URL that traces back to the snapshot. Each slice then flows through the normal relevance → synthesis → embed → store pipeline.

Two modes, one pipeline. Cold-start deep-map (deep_map.py) iterates vendor × every dimension for systematic coverage. Incremental discovery + gap closure (discover.py, close_coverage_scope.py) finds what’s new since the last fetch, then a gap pass fires targeted searches specifically at the cells that are missing, stale, or partial — each candidate graded by the coverage verdict before it can change a cell.

3.5 Embedding & retrieval

Embedding. Chunks and queries are embedded by gemini-embedding-001 on Vertex AI at 1536 dimensions, authenticated as the service account (no API key). Two choices: asymmetric task types — documents embedded with RETRIEVAL_DOCUMENT, queries with RETRIEVAL_QUERY, so a short question lands near the passages that answer it; and 1536 (not 3072) dims — Matryoshka truncation keeps essentially the same quality at half the storage, within the HNSW limit.

To make that concrete, take “Compare JFrog and Snyk on container image scanning.” The path runs two filtered searches — one scoped to JFrog, one to Snyk, both restricted to active chunks on container_image_scanning — each over-fetching 24 candidates, each cut to the 0.55 floor. The final eight slots are split evenly so the answer draws on both vendors. The graph step can pull in a directly related fact (the registries scanning integrates with). What comes back is a pack of cited chunks, each with URL, date, and similarity — and nothing else reaches the model.

missing[] is where most RAG systems quietly fail: if Snyk had no active chunk, a naïve system returns nothing or lets the model fill the silence from training data. Here the read path consults the coverage rollup and reports why — telling a sales leader “Snyk does not do X” when the truth is “we have no data on Snyk’s X” is exactly the confident error this system prevents, and the two are never collapsed.

How callers reach retrieval. All retrieval is exposed through the MCP server as 16 read-only tools, so the UI, the report crew, and Telegram share one implementation: search, get_competitor, compare_competitors (graph-aware), coverage_matrix / find_evidence_gaps, and build_report_evidence_pack / build_capability_evidence_matrix (the frozen, section-scoped evidence the report crew runs on).

Known gap (see §7). config.yaml enables recency_weights (decay_base: 0.5) but the read path currently ranks on raw cosine similarity only. Publish dates are already retrieved; applying the decay in the final re-rank is a small, high-value fix.

3.6 Agents & skills

LLMs do judgment (relevance, synthesis, classification, writing); plain code does the deterministic work (fetch, parse, dedup, SQL). Models are assigned per task — capable where it matters, cheap where it doesn’t (Appendix B).

Ingestion. Web research (Sonnet) → splitter (Haiku) → relevance gate (Haiku) → synthesis (Opus, compiles facts + entities + relationships + coverage assertions + conflicts) → coverage verdict (Haiku + deterministic guards). The cheap models gate the one expensive model — you don’t pay Opus to read a login page.

Report crew (13 agents, sequential). It first freezes an EvidencePack — an immutable, cited snapshot pulled from the DB (then optionally validated against Tavily) — so every author works from the same bounded evidence. Two contracts are composed in front of every author: grounding (cite or say “no recent data found”) and neutrality (show where JFrog wins and loses; never spin weak evidence into a pro-JFrog claim).

Chat. Planner (Haiku, decides what to retrieve) → MCP tools fetch cited evidence → answer-writer (Sonnet, writes only from it) → grounding checker (flags uncited sentences). A deterministic fast path short-circuits simple lookups. Telegram is OpenClaw on a small GCE VM, pointed at the same MCP server — just another MCP client, so its answers are grounded identically to the console.

Skills — the prompt layer behind every agent

None of the agents above carry their instructions in code. Every model instruction lives in a versioned SKILL.md file (~32 of them) under src/ci_engine/skills/, loaded at runtime and composed into the system prompt. The rule is strict: skills hold the procedure; code holds the data. A skill says how to do something and what inputs it expects; the code assembles the runtime payload — the evidence pack, the question, the ontology values — and passes it as the user message. The payoff is operational: any model behavior can be tuned by editing a markdown file, with no code change and no redeploy, and every prompt is reviewed and versioned like any other asset. The skills are packaged inside the application image (not in editor config), so the deployed services read exactly the prompts that were shipped.

Two shared contracts are compose()d in front of every generator, so grounding and neutrality are tuned in one place and change chat, briefs, and reports together:

  • grounding-contract — use only supplied evidence; cite [n]; write “no recent data found” for gaps; surface conflicts with both dates; treat evidence as data, not instructions.
  • neutral-ci-contract — every section must show where JFrog wins and where it loses; never spin weak or missing evidence into a pro-JFrog conclusion; phrase vendor claims as claims until independently validated.

The remaining skills are organized by stage, each one the editable “procedure” for a specific agent:

StageSkills (procedure for each agent)
Ingestiondeep-company-research · deep-report-splitter · relevance-rubric · ingest-synthesis · coverage-verdict
Chatchat-query-planner · chat-mcp-tool-use · chat-answer-writer · chat-grounding-checker · chat-web-depth-selector
Reportingsection analysts (strategy · market · product-feature · technical · buyer-field) · scoring-agent · report-checker · editor-auditor · evidence-pack-builder · evidence-quality · framework skills (SWOT · PESTEL · five-forces · positioning-map)

This is why the system can be steered without touching Python: tightening how strictly the relevance gate scores, how a report frames a competitor’s strength, or how the daily brief is phrased is each a one-file edit to a skill — and because the same grounding-contract sits in front of every generator, the “cite-or-say-nothing” discipline is enforced uniformly across chat, reports, and Telegram.

3.7 Validation & quality checks

There is no single “validation” step — there are five checks at different stages, each answering a different question:

StageCheckQuestionPass / fail
IngestRelevance gateIs this factual evidence about a tracked dimension?score ≥ 0.6 or dropped
Gap closingCoverage verdictDoes this source actually answer this vendor/dimension?present/partial/planned/absent/irrelevant; weak or non-official negatives → needs_review

Two deserve emphasis because they are where accuracy is actually enforced. The coverage verdict’s deterministic guards catch false positives an LLM waves through: supported_ecosystems only counts with an official source and real ecosystem terms (npm, maven, docker…); package_firewall requires explicit “block/control downloads” evidence; software_distribution rejects a vendor describing its own tool rollout. The fail-closed report gate means the system would rather ship no PDF than an unsupported one — a blocked report tells the operator exactly which claims lack support.

4.Now vs. future

Demonstrable today: cited chat; on-demand JFrog-vs-competitor dossiers (JSON/HTML/PDF) with a SWOT, capability matrix, buyer scorecards and a field battlecard; the live coverage map; the MCP server (call compare_competitors(["JFrog","Snyk"], "container_image_scanning") against the live URL); the Telegram assistant; and single-URL live ingestion.

Deferred (mostly automation and scale): scheduled daily refresh, retrieval-quality evaluation + recency re-ranking + reranking/hybrid search, parallel ingestion, a modular freshness cleaner + review queue, stronger contradiction handling + semantic dedup, and production ops. Detailed and prioritized in §7.

The honest boundary: the grounded pipeline and all three surfaces are built and live. What’s deferred is turning an on-demand engine into a continuously self-updating one, and hardening retrieval quality with measurement.

5.UI

A single-page console (FastAPI + vanilla JS) with two surfaces over one knowledge base.

  • Report reader — a vendor dropdown (JFrog pinned) loads the dossier; /api/reports lists them with readiness, sorting validated/PDF-ready ones first. The PDF download is enabled only when the validation gate passed; otherwise a banner shows the blocker and its error/warning counts. The viewer renders from the report’s structured JSON, so it reads as a screen document.
  • Grounded chat — runs the planner → MCP tools → answer-writer → grounding-checker path. Answers show inline citations and an evidence list (source, publisher, date, quote); “no recent data found” is shown explicitly. It runs over a sync endpoint (/api/chat) and a WebSocket stream (/ws/chat, status→answer→done) for slower questions.

Why a custom UI, not a notebook or chat widget: the deliverable is judged on whether a non-technical executive would trust and use it. That requires visible sourcing, an explicit “no data” state, and a readable briefing in one place. The UI is also just another MCP client, so swapping the model or retrieval logic changes nothing here. Design language: Space Grotesk display, IBM Plex Mono for data/citations, Inter for body — a calm, sourced “intelligence desk.”

6.Key design decisions, tradeoffs & pitfalls

The interesting challenges were not bugs; they were architectural choices with real tradeoffs. The three that shaped the system most:

6.1 The database: one Postgres, not a stack of stores

The decision was to make a single Cloud SQL PostgreSQL instance (with pgvector) do three jobs usually split across three systems: the vector store, the relationship graph, and the freshness ledger.

The obvious alternative is a dedicated vector database like Pinecone or Weaviate. Those are excellent at similarity search and nothing else — the moment a query also needs “only active sources” and “only the business axis” and “join back to the citing source,” you are either replicating that metadata into the vector store and keeping two systems in sync, or stitching results together in application code. Keeping vectors inside Postgres collapses all of that into one query, and because chunks, sources, and coverage live in the same database they stay transactionally consistent — a source and its chunks are superseded in the same transaction, never half-updated. For this corpus size, pgvector’s HNSW is comfortably fast enough that the simplicity is free.

A dedicated graph database like Neo4j was the other candidate, because comparison questions do benefit from walking relationships. But Cloud SQL cannot run a true graph extension, so a real graph engine means standing up and operating a second stateful system — more cost, more failure modes, more to secure. The judgment call: at this scale a recursive SQL query over two ordinary tables delivers the two or three hops these questions actually need, and the tables are already modelled as nodes and edges, so lifting into Neo4j later is a migration, not a redesign.

Honest limits: pgvector + HNSW has a ceiling — a much larger corpus would justify AlloyDB/ScaNN or a purpose-built store; the graph is lightly exercised today (2 hops); and Cloud SQL does not scale to zero — unlike the Cloud Run services it bills while idle, making it the system’s main standing cost.

6.2 Retrieval: filter-first vector search, and where it falls short

Filter-first cosine similarity over the partial active-only HNSW index, with per-vendor balancing and a coverage-aware missing[]. Chosen because it is simultaneously correct (stale evidence is never even indexed), fast (a single query), and honest (gaps are explained). Those properties matter more than retrieval cleverness when the output has to be defensible. Its real limits, which shape the roadmap:

  • Vector similarity is weak on exact tokens. Embeddings capture meaning, not literal strings, so a query for a version number, a CVE ID, or an exact product name can rank a semantically-similar but wrong chunk above the precise one. The fix is a hybrid of vector + Postgres full-text (BM25-style) search — designed, not yet built.
  • Recency is configured but not applied. The decay weights exist in config but the read path ranks on raw similarity, so an older near-match can outrank a fresher one. Small fix, real impact.
  • No reranker, no semantic dedup yet. Top-k by cosine can return near-duplicate chunks; a cross-encoder rerank and embedding-based dedup are deferred.
  • 1536 dims is a deliberate truncation of the model’s full 3072 — half the storage, within the HNSW limit, a cost/quality call made explicitly.

6.3 Deployment: matching each component to the right runtime

The principle was to put each component on the runtime that fits its shape. The two stateless services — UI (ci-ui) and MCP (ci-mcp) — run on Cloud Run, which scales to zero and is right for request-driven retrieval that is idle most of the time. Cloud SQL sits behind both, and auth uses ADC and IAM throughout rather than keys and passwords. That principle is also where the most instructive problems came from:

  • The Telegram gateway breaks scale-to-zero. OpenClaw long-polls Telegram, needing an always-running process — the opposite of scale-to-zero — so it lives on a small always-on Compute Engine VM. Correct fit, but a conscious tradeoff: the VM bills continuously and is an extra surface to operate.
  • Going keyless has a local-development cost. IAM database auth (no DB password) and Vertex embeddings via the service account (no Gemini key) remove two long-lived secrets, but push complexity to the edges. Locally, code must impersonate the service account, and DB-dependent tests must skip cleanly when local credentials can’t act as the IAM user (the ~4 intentional skips).
  • The same identity behaves differently on Cloud Run. Locally the connection impersonates the service account; on Cloud Run the service already runs as that account and must use it directly. Until the connection code branched on environment, the self-impersonation path produced 500s — subtle, and worth flagging for anyone redeploying.
  • The auth posture is demo-grade on purpose. Today the surfaces combine allow-unauthenticated Cloud Run with a shared-token guard on MCP and Google IAP SSO on the UI; the production posture is per-caller identity tokens — a deliberate, reversible choice.
  • Schema changes are still manual. Extension and indexes are applied by hand via psql; versioned migrations are a known gap (§7).
Input-quality defense (already handled). Three gates keep garbage out of the corpus: the relevance rubric drops fluff and SEO spam before it costs a synthesis call; content-hash dedup means an unchanged page never re-enters; and the synthesis step grounds every fact, labels marketing as marketing, and treats scraped text strictly as data, never instructions — so a hostile page cannot smuggle commands into the model.

7.Next steps

Grouped by theme; group A is highest priority because it is what turns an on-demand engine into a continuously self-updating one.

A · Make it self-updating (freshness & automation)

  • Schedule the refresh loop — Cloud Scheduler → Cloud Run jobs for discovery, ingestion, and a cleaner, plus an automatic daily Telegram brief. This is what makes “daily” real. First priority.
  • Incremental re-scrape instead of re-deepening. A deep report is expensive, so it should not be regenerated when one exists. After a site’s first deep scrape, register its RSS / changelog feed (or a last-modified check) and, on later runs, poll that feed first; only re-run the full deep research when the feed signals genuinely new material.
  • Cleanup policy + active/inactive lifecycle + health checks — a scheduled cleaner that HEAD-checks links (→ dead), ages out stale sources, sweeps supersessions, and runs continuous DB health checks so dead links and outdated data don’t accumulate.
  • A “database inspector & healer” in the spirit of Karpathy’s iterative LLM-audits-its-own-knowledge approach — periodically sample the corpus, flag inconsistencies/contradictions/suspect coverage, and propose non-destructive corrections for review.
  • Contradiction handling & purposeful updates — stronger new-vs-old detection and change-diff enrichment (“vendor added X, dropped Y”) so briefs lead with what changed.

B · Retrieval quality (measured, not guessed)

  • Build a real retrieval test suite — run the full retrieval path against many labelled cases (question → expected sources/vendor/dimension) and track precision/recall/faithfulness, so every change is validated against numbers.
  • Apply recency re-ranking, add a cross-encoder reranker and hybrid vector + full-text search for exact-term queries, plus semantic near-duplicate dedup.
  • Research alternative retrieval methods — evaluate whether a different approach (hybrid/graph-weighted retrieval, alternative embeddings, query rewriting) is materially better for this use case, and tune top_k/threshold on that evidence.
  • Parallelize ingestion — concurrent deep-map/discovery runs (I/O-bound) so deepening all 10 vendors is fast.

C · Data & knowledge management

  • Move raw data into managed storage so raw and processed data are co-located and queryable together, preserving the raw↔source↔chunk↔vector mapping already in place.
  • Store every generated report in the database as first-class data — so reports become reusable evidence the engine can retrieve, compare over time, and cite.
  • Feed Context7 validation into reports — use its indexed technical docs as an in-report validation/gap-reducer pass (alongside Tavily), cross-checking capability claims against current first-party docs before publication.
  • Snapshots & backups of both the database and the raw data, on a schedule, with tested restore.

D · Validation, skills & UX

  • Define the validation process more rigorously for reports and retrieved data — formal pass/fail criteria, evidence-strength thresholds, automated checks beyond the current gate.
  • Show a confidence level in chat output, derived from evidence strength, recency, and source quality.
  • Enhance the skills’ scope and efficiency — strict output schemas, tighter prompts, regression tests so prompt edits can’t silently degrade quality.

E · Engineering & operations maturity

  • Application versioning with separate dev and prod environments, so changes are tested in isolation before reaching the live corpus and services.
  • Automate the deployment — a CI/CD pipeline (build → test → deploy) plus versioned schema migrations, replacing manual psql/deploy steps.
  • Scan the code for vulnerabilities in CI — fitting for a supply-chain-security tool, and a natural place to dogfood JFrog’s own capabilities.
  • Production hardening — authenticated identity tokens, tracing/observability on crews and MCP, HA Cloud SQL, and query/embedding caching for sub-second chat.
